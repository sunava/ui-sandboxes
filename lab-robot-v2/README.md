# lab-robot-v2 — The Lab Robot (Tracy), rebuilt properly

Second iteration of the AICOR showcase UI. Scope: **only the first lab**
("The Lab Robot" — Tracy), copied from `../html-showcase/lab-robot.html` as a
working starting point. Goal: get the fundamentals right before adding the
other scenarios back.

## The knowledge base is now the REAL AICOR L2 ontology

The right-hand reasoning panel no longer runs a hand-written toy KB. It runs the
actual **AICOR L2 conceptual framework** — 300 classes, 310 subclass edges,
362 OWL restrictions, plus worked-example ABoxes (egg-crack, pour-milk-coffee,
fetch-milk-pipeline, …). Source: `~/Downloads/L2-conceptual-framework`.

**Tracy is now part of the KB too.** `tracy.urdf` is compiled into an ABox
(`tools/abox/tracy.yaml`, generated by `tools/urdf_to_abox.py`) that instantiates
the L2 body model: `tracy : Robot` → `tracy_body : RobotBody` with every URDF
link as a `Link`, every joint as a `Joint` (`connects` its two links,
`hasJointType`), the two Robotiq grippers as `EndEffector`, the camera as
`Sensor`, and a `Motor` (`actuatedBy`) for each independently actuated joint
(revolute, non-mimic). So `isa(I,'Joint')`, `isa(I,'EndEffector')`, etc. now
return Tracy's real parts — the ontology and the 3D robot share one description.

Two static Prolog files carry it, consulted into tau-prolog in the browser:

- `static/aicor-kb.pl` — **GENERATED** ground facts: the TBox (`class/1`,
  `subclass_of/2`, the `restriction*` facts, `disjoint/2`) and the ABoxes
  (`ex_individual/3`, `ex_triple/4`, `in_example/2`). Do not hand-edit.
- `static/aicor-rules.pl` — the portable rule / query layer on top:
  `subclass/2`, `superclasses/2`, `isa/2`, `constraint/4`, `triple/3`,
  `incompatible/2`. Written to the tau-prolog subset (no `aggregate_all`, no
  `discontiguous` prefix directive).

### Regenerating the KB

```bash
PYTHON=~/.virtualenvs/cram-env/bin/python tools/build-kb.sh   # needs rdflib + pyyaml
```

`tools/build-kb.sh` runs the framework's own compiler (`tools/okf_prolog.py`)
over the vendored `tools/aicor-ontology.ttl` + `tools/abox/*.yaml`, then strips
the SWI-only `:- discontiguous` lines so tau-prolog can parse the output.

To (re)generate Tracy's ABox from the URDF:

```bash
tools/urdf_to_abox.py static/tracy.urdf --name tracy --out tools/abox/tracy.yaml
# then rebuild the KB (build-kb.sh picks up every tools/abox/*.yaml)
```

## Run

```bash
cd ~/ui-sandboxes/lab-robot-v2
python3 server.py            # http://localhost:8711  (add a port arg to change)
```

No build step at runtime, no internet — everything is vendored.

## Layout

- `index.html` — the single page (was `lab-robot.html` in html-showcase)
- `static/kb.js` — loads the two `.pl` files into tau-prolog; defines the
  worked-example catalogue + query presets
- `static/main.js` — glue: derives the graph from the ontology (class taxonomy
  ↔ per-example ABox), the query panel, node-click explanations, the example strip
- `static/graph.js` — vis-network renderer (TBox classes / ABox individuals)
- `static/robot.js` — Three.js r128 + urdf-loader; renders `static/tracy.urdf`
  (UR10e, Robotiq 2F-85, TraceBot bench). **Ambient only** — the real ontology
  is domain-general, so the robot is no longer driven per workflow step.
- `static/trajectory.json` — recorded pycram+giskardpy motion for "Play robot motion"
- `tools/` — the KB build pipeline (compiler, ontology TTL, ABox YAMLs, build script)
- `server.py` — dependency-free static file server
